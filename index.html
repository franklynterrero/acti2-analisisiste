1. ¿Qué son los Modelos de Estimación?

Los modelos de estimación son metodologías y herramientas utilizadas para predecir diversos aspectos de un proyecto, como su duración, esfuerzo, costo y recursos necesarios. En el contexto del desarrollo de software, estos modelos permiten a los gestores de proyectos y equipos de desarrollo realizar proyecciones informadas sobre el tiempo y los costos implicados en la creación de un sistema de software, ayudándoles a planificar y tomar decisiones estratégicas de manera más precisa.

2. Diferentes Modelos de Estimación Utilizados en la Industria del Desarrollo de Software

Modelo COCOMO (Constructive Cost Model)

El modelo COCOMO es un método de estimación de costos de software desarrollado por Barry Boehm en 1981. Este modelo utiliza una fórmula basada en el tamaño del software, medido en líneas de código (LOC), para predecir el esfuerzo, tiempo y costo de desarrollo. Existen tres niveles de COCOMO:

  • COCOMO Básico: Proporciona una estimación rápida basada en el tamaño del software.
  • COCOMO Intermedio: Incluye factores de costo que ajustan la estimación según características del proyecto, como la experiencia del equipo y las herramientas de desarrollo.
  • COCOMO Detallado: Añade un análisis más detallado de los componentes del proyecto y sus interacciones.

Método de Puntos de Función

El método de Puntos de Función (Function Points) es una técnica de medición desarrollada por Allan Albrecht en los años 70. Este modelo mide la funcionalidad del software desde la perspectiva del usuario, evaluando las entradas, salidas, consultas, archivos lógicos y interfaces externas. Los puntos de función son luego ajustados por factores de complejidad técnica y ambiental.

Comparación

  • Enfoque:
    • COCOMO: Se centra en las líneas de código y factores de costo relacionados con el esfuerzo de desarrollo.
    • Puntos de Función: Se centra en la funcionalidad entregada al usuario y no depende del tamaño del código.
  • Adecuado para:
    • COCOMO: Proyectos donde se puede estimar el tamaño del software en líneas de código desde etapas tempranas y donde se dispone de datos históricos de proyectos similares.
    • Puntos de Función: Proyectos donde la funcionalidad desde la perspectiva del usuario es clara y se requiere una estimación temprana sin necesidad de conocer el tamaño del código.

3. ¿A qué llamamos la estimación de un proyecto de software?

La estimación de un proyecto de software se refiere al proceso de prever el esfuerzo, tiempo, costo y recursos necesarios para completar un proyecto de software. Este proceso incluye la identificación y evaluación de tareas, la asignación de tiempos y recursos, y la predicción de posibles riesgos y contingencias.

4. Principios Esenciales para el Desarrollo de Sistemas

  • Modularidad: División del sistema en módulos más pequeños y manejables.
  • Cohesión: Grado en que los elementos dentro de un módulo están relacionados entre sí.
  • Acoplamiento: Grado de dependencia entre módulos; idealmente, debe ser bajo.
  • Abstracción: Ocultación de los detalles de implementación y exposición solo de las funcionalidades esenciales.

Ejemplo de Aplicación de la Cohesión

En el desarrollo de un sistema de gestión de inventario, un módulo de "Gestión de Productos" debe contener todas las funciones relacionadas con los productos (como agregar, actualizar y eliminar productos) sin incluir funciones no relacionadas como el procesamiento de pagos.

5. Partes Interesadas en un Proyecto de Desarrollo de Software para una Aplicación Móvil

  • Clientes:
    • Roles: Proporcionan requisitos y feedback.
    • Responsabilidades: Definir necesidades y validar entregables.
    • Expectativas: Alta calidad, usabilidad y cumplimiento de requisitos.
  • Equipo de Desarrollo:
    • Roles: Programadores, diseñadores, testers.
    • Responsabilidades: Construir y probar la aplicación.
    • Expectativas: Claridad de requisitos y soporte técnico.
  • Gerente de Proyecto:
    • Roles: Planificación y seguimiento del proyecto.
    • Responsabilidades: Asegurar la entrega a tiempo y dentro del presupuesto.
    • Expectativas: Cumplimiento de plazos y calidad.
  • Usuarios Finales:
    • Roles: Utilizan la aplicación.
    • Responsabilidades: Proporcionar feedback.
    • Expectativas: Facilidad de uso y rendimiento.
  • Stakeholders Financieros:
    • Roles: Inversores o patrocinadores.
    • Responsabilidades: Proveer financiamiento.
    • Expectativas: Retorno de inversión y eficiencia de costos.

6. Plan de Proyecto para un Sistema de Gestión de Inventario

Fase 1: Iniciación

  • Actividades: Recolección de requisitos, análisis de viabilidad.
  • Dependencias: Ninguna.

Fase 2: Planificación

  • Actividades: Definición del alcance, planificación del proyecto, asignación de recursos.
  • Dependencias: Iniciación.

Fase 3: Diseño

  • Actividades: Diseño del sistema, diseño de la base de datos.
  • Dependencias: Planificación.

Fase 4: Desarrollo

  • Actividades: Codificación, integración de módulos.
  • Dependencias: Diseño.

Fase 5: Pruebas

  • Actividades: Pruebas unitarias, pruebas de integración, pruebas del sistema.
  • Dependencias: Desarrollo.

Fase 6: Implementación

  • Actividades: Despliegue, capacitación de usuarios.
  • Dependencias: Pruebas.

Fase 7: Mantenimiento

  • Actividades: Soporte y mantenimiento, actualizaciones.
  • Dependencias: Implementación.

7. Normas y Estándares en el Desarrollo de Software

  • Estándar de Nomenclatura: Uso de nombres descriptivos para variables y funciones. Ejemplo: En lugar de var1, usar cantidadProductos.
  • Documentación del Código: Cada función debe tener comentarios explicativos sobre su propósito y uso. Ejemplo: /** Esta función calcula el total de inventario **/.
  • Control de Versiones: Uso de sistemas como Git para gestionar cambios en el código. Ejemplo: Crear ramas para nuevas funcionalidades y realizar merges solo después de la revisión de código.
  • Revisión de Código: Implementar revisiones de código regulares por pares para garantizar la calidad. Ejemplo: Antes de fusionar una rama, otro desarrollador debe revisar y aprobar los cambios.
  • Pruebas Automatizadas: Implementar pruebas unitarias y de integración. Ejemplo: Escribir pruebas automatizadas para verificar que una nueva funcionalidad no rompe las existentes.

Al seguir estos estándares, se garantiza que el código sea consistente, comprensible y mantenible, lo que facilita el trabajo en equipo y la calidad del software.
